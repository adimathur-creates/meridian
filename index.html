About this build

A working product-judgment artifact for the Investment Execution PM role — built to show I can own execution outcomes, not just run sprints.

THE PROBLEM

Investment execution fails quietly: a debit posts but funds don't realize before NAV cut-off, a payment can't be matched to an order, an allotment file disagrees with the placed value. Each is money in limbo and a client who can't be told the truth.

THE DESIGN

Every order answers three questions fast: where is the money, what must happen next, what can we safely tell the client. Two invariants hold the system honest — no placement before realization, no client NAV before allotment.

THE JUDGEMENT

Exceptions are ranked by money-at-risk against the cut-off clock, not by age — because a ₹6.5L order 90 minutes from cut-off matters more than a ₹10k one aged 28h. Operator actions are the recommended next move, not raw data.

Assumptions & scope

  • Built on synthetic orders and publicly-known Indian MF mechanics: BSE StAR MF / RTA flows, 3:00 PM equity NAV cut-off, T+1 allotment, NACH / UPI mandates.
  • Not a production system and uses no proprietary data — a product-judgment artifact, not an integration.
  • Reconciliation is modelled as a three-way match: bank debit ↔ order value ↔ allotment value; any delta opens a break.
  • State is live and in-memory: resolving an exception updates the overview, the queue, and the lifecycle view together.
